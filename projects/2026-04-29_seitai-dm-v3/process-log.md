# 整体院DMハガキ制作 — Process Log

> summary: 整体院DMハガキ制作 — Claude DesignでのDraw・Tweaks初回検証とAI協働パターンの記録

> このログは、AI（Claude Design）と人間の協働プロセスそのものを「LLMに再投入できる資産」として残すための記録です。
> 単なる成果物の履歴ではなく、**「曖昧な感覚をAIがどう言語化したか」「人間の非言語入力（Draw）にAIがどう推論で応答したか」** の流れを構造化しています。

### 1-4. プロセスの全体像（フェーズ別）

```
[フェーズ0] 立ち上げと最初の質問群
        ↓ 「深夜なのでターゲットだけ指定、他は任せる」
[フェーズ1] 仮定駆動の初稿（和モダン3案 + Voice/Warmth/Emphasisの第1世代Tweaks）
        ↓ リセットして質問群を再提示
[フェーズ2] 部分回答による方向転換（パステル × ワーキングマザー × 紹介主軸）
        ↓ 「重なってる、金額削除」
[フェーズ3] テキスト指示による構造修正（abs → flex縦積み、金額削除）
        ↓ 非言語入力フェーズ開始
[フェーズ4] Drawによる装飾・配置の対話的進化（花リース化 → はみ出し修正 → ヘッドライン統合）
        ↓ 「『初回』は紹介者には正確じゃない」
[フェーズ5] コピー精度の対話（既存顧客と新規友人の文脈差をどう扱うか）
        ↓ AIの環境理解の質問
[フェーズ6] メタ対話（編集モード・JSX言語・Comment/Drawの使い分け）
        ↓ ログ化の指示
[フェーズ7] プロセス記録の構造化（本ファイル）
```

### 1-5. このログの目的（ユーザー本人の言葉）

> 「最新のAIが『これまで人が"センス"と呼んで言語化できないものとして扱ってきた領域を、AIが的確な質問で引き出せるようになった』『ユーザー側が言語化できてない感覚的で曖昧な指示を、AIが推論を立てて〈そうそう！こういう感じ！〉といった風にビジュアルまたは推測に基づく幾つかの選択肢として言語化できるようになった』という体験を得る為。
>
> やり取りを記録し、激しく変化していくAI技術の中にあっても容易に陳腐化しない『繰り返せるパターン』を確立すること、その記録じたいをLLMに再投入できる資産とすること。」

このため本ログは、成果物の単なる履歴ではなく **対話のパターンと推論の往復を構造化** することを優先しています。

## 2. 非言語・ビジュアル指示ログ

### 2-1. Draw指示ログ

人間が「言葉にしづらい」もしくは「言葉にすると遠回りになる」指示を、画面上の落書き（Draw機能）でAIに渡した記録。AIはこれをマルチモーダル入力として解釈し、必要に応じて確認質問を返した。

時系列で5枚。それぞれ **画像 / 解釈 / 確認質問の有無 / 実装結果 / 学び** の形で構造化している。

---

#### Draw 1 — 「枠を花模様にし」

| 項目 | 内容 |
|---|---|
| ファイル | `assets/draw-1d023cfc.png` |
| フェーズ | フェーズ4の起点 |
| 画像内容 | A案オファーカード上部から右上にかけて赤い丸で囲み、別途付箋に「悪を花模様にし」のテキスト |
| AIの解釈 |（1）「悪」は「枠」の漢字変換ミスと推察し自動補正<br>（2）赤丸の位置から、A案オファーカードの**外周枠線**を装飾化したい意図と判断 |
| 確認質問の有無 | **あり**。解釈幅を残すため2軸で4択ずつを提示：<br>① 装飾密度（a) 隅だけ / b) 全周リース風 / c) バッジ周囲 / d) おまかせ）<br>② モチーフ（a) 桜・梅などの和風 / b) 5弁の野花 / c) ボタニカル葉実 / d) おまかせ） |
| 人間の回答 | **「b と b」** （= 全周リース風 × 5弁の野花） |
| 実装 | `<FloralBorder>` `<Flower5>` `<Leaf>` の3コンポーネントを新規追加。SVGで5弁花と葉を周長配置。Mood設定で密度連動 |
| 学び | **Drawの曖昧さは「2軸×4択」で言語化できる**。AIが選択肢を並べることで、ユーザー側の意思決定コストが大幅に下がる。typo自動補正もコンテキストから可能 |

---

#### Draw 2 — 「枠線に合わせて」

| 項目 | 内容 |
|---|---|
| ファイル | `assets/draw-c4e3203c.png` |
| フェーズ | Draw 1直後の修正フィードバック |
| 画像内容 | 花リースが実装されたカードの**右下角**を赤丸で囲み、付箋に「枠線に合わせて」のテキスト |
| AIの解釈 | 花がカードの実際の境界からはみ出している箇所の指摘。コード側の原因として、`<FloralBorder>`に`w/h`を**固定値**で渡していたため、カード実寸と乖離していると判断 |
| 確認質問の有無 | **なし**。原因が構造的に特定できたため確認不要と判断 |
| 実装 | `ResizeObserver`で親要素の実寸を取得し、辺の長さに比例して花を再配置するロジックに書き換え。border-radiusに沿う四隅配置も追加 |
| 学び | **「位置の赤丸」+「短い言語ヒント」だけで、AIはコード上の構造的不整合（固定値ハードコード）まで遡れる**。微調整に見える指摘が、実は実装方針そのものの修正を要求しているケース |

---

#### Draw 3 — バッジ装飾の試し書き

| 項目 | 内容 |
|---|---|
| ファイル | `assets/draw-c9eaae2b.png` |
| フェーズ | Draw 1〜2の周辺、明示的なフォローアップなし |
| 画像内容 | オファーカード上部のバッジ部分から外側へ向かう矢印。付箋に「バッグをもっと華やかな」（→「バッジを」のtypo） |
| 状態 | 試し書きとして残された断片。実質的にはDraw 1の花リース実装に意図が包含された可能性が高い |
| 学び | **すべてのDrawが明示的アクションを引き起こすわけではない**。試し書き・思考整理として機能することもある |

---

#### Draw 4 — 「反応する」断片

| 項目 | 内容 |
|---|---|
| ファイル | `assets/draw-94c944f3.png` |
| フェーズ | フェーズ間の試し書き |
| 画像内容 | 宛名面下部に「反応し」「反応する よ」のような付箋。具体的な要素への矢印や赤丸はなし |
| 状態 | 思考メモ的な断片。明示的な指示にはならず |
| 学び | Drawはユーザー自身の **思考を可視化する場** としても使われる。AIへの指示ではなく、自分用のメモになることもある |

#### Draw 5 — ヘッドラインとカードの統合（誤読 → 訂正）

| 項目 | 内容 |
|---|---|
| ファイル | `assets/draw-f22b35fb.png` |
| フェーズ | フェーズ4の最大の往復 |
| 画像内容 | 「お友だちと、ふたりで通う春。」のヘッドラインから、カード内の「ご紹介者さまも、お友だちも／どちらも初回が半額に。」へ向かう赤い矢印 |
| AIの初回解釈（誤） | 矢印が **下方向** を向いていることから、ヘッドラインをカード内へ移動させる指示と読んだ |
| 人間の訂正 | 「**逆**です。矢印の先のテキストを上書きする＝削除。そしてカード内の『あなた』『お友だち』をもう少し大きく目立つように」 |
| AIの再解釈（正） | （1）元ヘッドライン削除<br>（2）カード内メッセージを元ヘッドライン位置に移動<br>（3）カード内はバッジ＋DuoMarkだけのシンプル構成に<br>（4）DuoMarkを 42px → 64px に拡大 |
| 確認質問の有無 | **誤読時は実装に踏み切らず提案で止めた**ことで訂正余地を残した |
| 学び | **手書き矢印は方向の解釈が反転しうる**。AIは矢印を含むDrawを受け取った際、実装前に矢印の意味を言語で確認するのが安全 |

### 2-2. Tweaks操作ログ

Tweaksは「AIが事前に仕込んだパラメータをユーザーがUIから直接いじれる」機構。今回のセッションでは2世代にわたって設計され、初稿コンセプト変更とともに作り直されている。

#### Tweaks 第1世代（フェーズ1：和モダン3案時）

| 軸 | 選択肢 | 役割 |
|---|---|---|
| Voice 声色 | ご挨拶 / 心配り / 感謝 / オファー前面 | メインコピーの人格そのものを切替（A/B/C全案に同時適用） |
| Warmth 距離感 | フォーマル ↔ 親密 | 紙色とアクセント彩度をスライダーで連続変化 |
| Emphasis 主役 | 言葉 / バランス / 数字 | ジャンプ率を再設計 |

**特徴**：3案すべてに同時に適用される横断的なレバーとして設計。

#### Tweaks 第2世代（フェーズ2以降：パステル3案時）

| 軸 | 選択肢 | 役割 |
|---|---|---|
| Palette | Rose & Mint / Cream & Lavender / Yellow & Sage | 3つのパステルパレット |
| Mood | すっきり / ふんわり / ぷち華やか | 装飾の密度・角丸の強さを連動 |
| Duo Emphasis | 控えめ / バランス / しっかり | 「ふたり半額」の強調レベル |
| 送付月 | 1〜12月のスライダー | キャンペーン期間表記が動的に再計算 |

#### 抽出される再現可能パターン

1. **「全案同時適用」が比較設計の核** — Tweaksを変えると3案すべてが同時に変化し、条件×案のマトリクス比較が成立する
2. **離散プリセット + 連続スライダーの混在** — 離散はコンセプト切替、連続は動的計算が絡む数値に使い分ける
3. **コンセプト変更時はTweaksも再設計** — 前世代の軸を引き継ぐと不要な自由度が残る

### 2-3. Commentピン留め（参考）

このセッションでは Comment 機能は使われなかった。AIから提示された住み分け：

- **Edit** — テキストを直接打ち替え（自分の手で）
- **Tweaks** — 用意されたパラメータを操作（自分の手で）
- **Comment** — ピン留めして文章で指示（人間にもAIにも）
- **Draw** — 絵で感覚的に指示（主にAIに）
- **チャット** — 自由記述で指示（主にAIに）

「Drawで位置を指す → チャットで補足言語化」で十分機能したためCommentは出番がなかった。**ピン位置の特定が必要かつ言語表現も必須なケースで初めてCommentが真価を発揮する** — 次セッションで検証すべきパターン。

## 3. このセッション自体が示したパターン（メタログ）

> ここに記録されている内容は、Claude Designでのデザイン制作プロセスだけでなく、
> 「AIとの役割分担をどう設計するか」という問いへの実験でもあった。

### 3-1. Claude.aiとのやり取りで発生した失敗パターン（同日）

| 失敗 | 原因 |
|---|---|
| AIが12章立てをアドリブで生成 | 初手で章構成を渡さなかった |
| 2章と3章の内容が重複 | 各章の定義を事前に確定しなかった |
| 見えないファイルへの統合指示 | コンテキスト共有範囲を確認しなかった |
| 「コードブロックで出力」の無意味な繰り返し | AIがCanvasの存在を忘れて定型句を返した |
| チャット跨ぎでの文脈消失 | Claude Designのチャット単位の独立性を前提にしていなかった |
| process-log.md生成リトライでDesign使用量77%消費 | 重い処理をOpus 4.7で何度も失敗させた |

### 3-2. 抽出されるパターン（ツール・バージョン非依存）

1. **AIに渡す前に「構成・範囲・担当の切り分け」を一枚に書く**
2. **「AIが生成する部分」と「人間が書く部分」を先に分離する**
3. **コンテキストの共有範囲を毎回確認する**（チャット・Canvas・別チャットはそれぞれ独立）
4. **長文化は矛盾の温床** — 短く区切って確認する方が速い
5. **重い処理の設計はDesignに入る前にSonnetで済ませる**

### 3-3. 再現性の条件

このログが「成功法則本」と異なる条件は、**失敗と判断根拠が記録されている**点にある。
「こうしたら成功した」ではなく「この条件でこう判断し、その根拠はこれ」が書かれていれば、
ツールが変わっても・AIが進化しても、判断の枠組みは転用できる。
