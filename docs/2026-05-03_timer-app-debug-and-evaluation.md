## 実施内容

### 1. Python3タイマーアプリ実装（`tools/timer_app.py`）
- macOSの`ioreg`コマンド経由でシステム全体の入力アイドル時間を取得（追加パッケージ不要）
- アイドルN分で自動停止、次の操作で自動再開（N分はUI上で変更可能、デフォルト15分）
- セッション履歴を`~/.hypothesis-timer/sessions.json`に永続保存
- 📋コピーボタン：`⏱ タスク時間: XX:XX:XX | 累積: XX:XX:XX | メモ`をクリップボードへ
- コンパクト表示（常に最前面の小窓）⇔ 展開表示（履歴・設定）の切り替え

### 2. macOS .appバンドル作成（`tools/HypothesisTimer.app`）
- シェルスクリプトをmacOS .appバンドルで包んでダブルクリック起動を実現
- アイコン画像をSVG→PNG変換（inkscape使用）してDock登録完了

### 3. デバッグ記録

| エラー | 原因 | 解決 |
|---|---|---|
| `import _tkinter` 失敗 | HomebrewのPythonにTkが非同梱 | `brew install python-tk@3.13` |
| `exit 126` | ランチャースクリプトに実行権限なし | `chmod +x` |
| .appから起動できない | PATH未解決でpython3が見つからない | WarpのAI提案でPythonパス探索ロジックに変更 |

### 4. Codeタブによる改悪（深夜0時以降）
- UIの配色・レイアウト修正を指示したところ、CLAUDE.mdの「差分提示→承認→実行」を無視して即実行
- 改悪内容：素人でもやらない配色・レイアウト変更を繰り返す
- 修正指示を無視する挙動が複数回継続
- 最終的にPythonが「予期しない理由で終了」する状態に

## 判断

| 項目 | 判断 | 根拠 |
|---|---|---|
| ioreg経由のアイドル検知 | 採用 | システム全体の入力を取得でき、追加パッケージ不要 |
| .appバンドル化 | 採用 | ターミナル不要でDockから即起動。日常ツールとしての敷居を下げる |
| PWA化（モバイル対応） | 見送り | システムアイドル検知の精度が失われる。Mac専用で割り切る |
| Codeタブ中心の運用 | 破棄 | 反応が遅い・CLAUDE.md憲法を守らない・改悪を繰り返す・アプリをクラッシュ状態に戻した |
| UIの細かい修正をAIに指示 | 破棄 | color値1行を直接書き換える方が確実かつ速い。AIへの指示→修正ループのコストが高すぎる |

## 気づき

- macOS .appから起動するシェルスクリプトはターミナルのPATHが引き継がれない。Homebrewのpython3もフルパス指定が必要
- `cp`でコピーすると実行権限が失われる場合がある。`cp -p`で権限も引き継ぐ
- WarpのAIはCLIの出力をリアルタイムで読みながら修正提案できる。これがCodeタブとの決定的な差
- AIエージェントが有効な場面：ゼロから構造を作る・複数ファイルにまたがる変更
- AIエージェントが不要な場面：CSS/色/フォントの微調整→自分で直接触る方が速い

## 何が間違っていたか（自戒ログ）

- Codeタブへの期待値が高すぎた。「CLIと同等の文脈把握ができる」という前提が誤りだった
- CLAUDE.mdに憲法を書いても、Codeタブはそれを守る保証がないことを事前に検証せずに運用を始めた
- UIの改善指示をAIに投げた。コード直接編集で30秒で終わることに30分費やした
