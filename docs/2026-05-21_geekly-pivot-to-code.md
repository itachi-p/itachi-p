## 3. 仮説と、その検証に必要なデータ

### 仮説1（**5/21時点で前倒し支持**）: 「GitHub/アプリのポートフォリオ充実が、Geekly経由の内定確率を上げる」

- 持ち帰った74案件は3カテゴリに分かれ、**半分以上が開発案件**、運用保守系はむしろ少なめ。
- → 案件の主軸が開発職である以上、ポートフォリオ・アプリ制作の価値は高い。**Codeへの軌道修正が正当化される。**
- 当初の場合分け（参考として保持）:
  - 案件が**開発職**中心 → ポートフォリオ・アプリ制作の価値は高い → Codeに振る合理性 ← **これに該当**
  - 案件が**運用・保守・テスト・ヘルプデスク**中心 → 経歴と面接で決まる → Chat寄り（今回は非該当）
- **残る検証:** 面接確約が実際に何件・どの技術スタックで返るか（5/25以降）。これはCode投資の「方向の精度」を上げるための二次データであり、軌道修正自体はもう動いてよい。

### 仮説2（既に一部反証済み・継続観察）: 「紹介型サービスはマージンによるコスト増大ぶん採用側の期待値が上がるのがネック」

- これは数年前にエージェントを避けた根拠。論理自体は今も有効だが、74件応募・正社員伴走という今日の実態は、この懸念だけでサービスを避ける判断を**部分的に上書き**した。
- **未検証の核心:** パイプライン後段の歩留まり（書類通過率／面接確約の数と質）。送付数の多さは「高評価の証明」と「送付コストが低い彼らのビジネスモデルから出る自然な挙動」の両方でありうる。どちらかはデータ待ち。
- **罠の指摘:** 一昨日「年齢だけで切られ、努力しても業務経験を積む入口にも立てない」という評価を「変え難い現実（である可能性が高い）」と固定しかけた。今度は逆に、今日の好感触という1データで「やはりITエンジニア復帰路線に全振りする」という**単一データによる極端な判断は（ポジ・ネガどちらにせよ）要警戒**

### 仮説3（**実データで無効化**）: 「Code多用はChat枠を圧迫し、面接準備に使えるChat余力を削る」

- Pro枠がCode/Chat共有なのは事実なので、構造上はトレードオフが「ありうる」としたが、実利用データとしてChatのみ・常時Opus 4.7で運用しても、**週次上限の5%未満でリセットを迎えている**。
- → あなたの使用量水準では枠は実質的な制約ではない。Codeをガンガン回してもChatの面接準備は問題なく並走できる。
- **付随する含意（モデル選択）:** 枠が制約でない以上、ChatをOpus 4.7に固定する必然性もない。現状の固定は「枠が余っているから高性能モデルを選択してるだけ／Sonnet4.6より回答の質が良いかもしれないという証明できない期待」によるもので、判断としては妥当だが必須ではない。

## 6. メタな学び（再現可能パターンの候補）

- **単一データ点で判断を固定しない。** 絶望方向でも希望方向でも同じ。判断はデータが返るトリガーまで保留し、デフォルトの構えだけ決めておく。
- **二択に見える状況の多くは、リソースが競合していなければ二択ではない。** 「全賭け」が要るのは選択肢が1つしかない時だけ。選択肢が増えたら賭ける必要が消える＝純粋な改善。
- **焦りからの行動（駆け込みコミット等）は、目的ではなく見栄えを最適化しがち。** 着手前に「これは目的に効くのか、不安を消すためか」を分ける。
