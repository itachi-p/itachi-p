# AIツール統合前提の更新ログ

## 文脈

projects配下のマインドマップPNG静画を編集可能形式に変換する作業。Claude（本対話）に方針を問うた際、「PNG→構造データへの自動変換は実質不可能、Markdown等の中間ファイル経由が必須」という回答を得た。これに従いMarkdown中間ファイルを生成。

しかし並行してXmind公式WebサイトのAI機能（Xmind AI）に直接PNGを投入したところ、`.xmind`形式へ一気に変換完了。AI回答の前提が古かったことが確認された。

## 視点設定

- 顧客視点：作業時間とツール選定の合理性
- 経営視点：「再現可能な戦略の型」（README.mdの根本目的）に対する判断フレームの妥当性

## 不便・違和感の言語化

| 観点 | 内容 |
|---|---|
| 直接的損失 | Markdown中間ファイル生成の作業ステップが冗長だった |
| 構造的論点 | AIへの問いに対する回答の前提自体が古びるリスク |
| 一般化可能性 | 「AIに聞く」と「サービス側AIを試す」を比較検討する習慣の欠落 |

## 仮説整理

### Why（なぜ起きたか）
Claudeの内部知識は2026年1月時点で停止。Xmind AIによる画像直接変換などサービス側AI内包の進展は学習データに反映されていない。回答自体は内部整合的だが、前提が古い。

### Who（誰の問題か）
- 一次的にはAIを使う側（プロンプトの設計者）
- 二次的にはAI回答に依存する判断プロセス全般

### What（何が起きていたか）
2024〜2025年は「AIが生成」「ユーザーが調整」のモデル。2026年は「サービス側にAI機能が内包され、エンドツーエンド実行する」モデルへ移行中。Genspark Claw（2026年3月発表）が「AIエージェント」レイヤーの代表例。Claude Code、Cowork等も同じ潮流。

## 検証

| 検証項目 | 結果 |
|---|---|
| Xmind AIによる直接変換 | 成功（`.xmind`ファイル生成済み） |
| Claude回答の前提古さ | 確認（中間ファイル必須の前提が崩れている） |
| 業界トレンドの裏付け | 複数ソースで「AI generation → AI execution」への移行を確認 |

## 判断

### 採用
1. **AIに問う前に、対象領域のサービス側AI機能を一次調査する**手順を判断フレームに追加
2. Markdownの位置づけ：AI対話・人間可読・汎用性のバランスから、中間ファイル形式として引き続き有用と評価
3. 運用パターン：Claude等で静画生成 → 専門ツール側AIで構造データ化 → 再度Claudeへ投入、のループを基本形に

### 保留
- Obsidian導入：現状のdocs/規模では関係性可視化ニーズが顕在化していない。不便を感じてから検討
- Xmindアプリのローカルインストール：Web版で完結する作業範囲なら不要

### 破棄
- 「AIに聞いた回答の前提は最新である」という暗黙の信頼

## 学習

### 一般化できるパターン
1. **AI回答の前提自体が古びる**：回答の内部整合性ではなく、前提となるツール環境・サービス動向の鮮度を疑う
2. **判断フレームの更新コストは個別ツール知識の更新より高い**：「中間ファイル必須」のような構造的前提は明示的にメンテナンスしないと残り続ける
3. **AIツール競争軸の移行**：単体性能 → 組み合わせの一気通貫実行（agentic AI）。ツール選定時の評価軸として「他サービス・APIとの統合性」「エンドツーエンド実行能力」を含める

### ロスの記録
- 中間ファイル生成のためのMarkdown化作業：約1往復分のAI対話コスト
- 回避可能だったが、回避コスト（事前調査）も発生するため、純粋なロスとは言えない
- ただし、今後同種のロスを防ぐには「サービス側AI機能の一次調査」を習慣化する必要あり

## 成果物比較：同一PNGから3経路で生成

本件の検証過程で生成された3つの成果物。同じ元データ（PNG）から異なる経路で生成され、それぞれ異なる特性を持つ。経路と特性の比較記録。

### 3つの成果物

| # | ファイル | 生成日 | 生成手段 | 形式 |
|---|---|---|---|---|
| 1 | `projects/thinking-tools/mindmaps/2026-05-17_disability-employment-current-state.png` | 5/17 | 前のチャットで生成（経緯詳細は別途） | PNG静画 |
| 2 | `projects/thinking-tools/mindmaps/2026-05-18_disability-employment-current-state.md` | 5/18 | Claudeが①の画像を読み取ってMarkdown化 | Markdown |
| 3 | `projects/thinking-tools/mindmaps/2026-05-18_disability-employment-current-state.xmind` | 5/18 | Xmind AIが①のPNGを直接変換 | Xmind独自形式 |

### 経路の違い

```
①PNG ─→ Claude（画像読取→構造抽出→Markdown生成）─→ ②Markdown
①PNG ─→ Xmind AI（画像→.xmind直接変換）─→ ③Xmindファイル
```

②は人間可読・編集容易・Git管理向き。③はXmindでの視覚的編集向き。両者は競合ではなく、用途が異なる。

### 経路ごとの特性比較

| 観点 | ②Markdown経由 | ③Xmind AI直接変換 |
|---|---|---|
| 生成までの手数 | 多い（AI対話で構造を起こす） | 少ない（PNGアップロードのみ） |
| 中間表現の汎用性 | 高い（他ツールにも展開可） | 低い（Xmindに固有） |
| 人間による構造検証のタイミング | 生成時に同時に行える | 変換後に別途必要 |
| Git管理との親和性 | 高い（diff可視） | 低い（バイナリ） |
| 内容修正のしやすさ | 高い（テキスト編集） | 中（GUI操作） |
| AI再投入時の扱いやすさ | 高い（構造化テキスト） | 低い（事前にエクスポート必要） |

### ファイル名規則の検討

3ファイルの日付が `2026-05-17` と `2026-05-18` に分かれている。これは「ファイル名の日付は実生成日とする」というルールに従った結果。元PNGは5/17に生成され、Markdown/Xmindは5/18にそれぞれ別経路で生成された。

slugは3つとも `disability-employment-current-state` で揃え、拡張子と日付の差で識別する。これにより「同一テーマの異なる成果物」が一覧で識別可能になる。

→ このルールはREPO_STRUCTURE.mdに反映予定。

### 学習

- 同一インプットから複数経路で出力を生成し、特性を比較する作業自体に検証価値がある
- 単一経路に固執すると、その経路の前提（中間ファイル必須など）に気づきにくい
- ただし全ての作業でこれをやるとコスト過大。「判断フレームに疑問を持った時」に限定して適用する
